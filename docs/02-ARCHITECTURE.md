# 系统架构

## 定位

AI Dev Team 是一个效率倍增器，让一个资深工程师拥有一支完整开发团队的产出能力。支持个人自部署使用，也支持 SaaS 模式服务外部用户（参见 [22-SAAS_ARCHITECTURE.md](22-SAAS_ARCHITECTURE.md)）。

## 核心设计原则

### 0. 基础设施层 - 资源池化、控制面与数据面分离

```
系统架构分为控制面和数据面两部分：

控制面（本系统）：
  Dispatcher + Web UI + AI 编排 + 任务调度
  → 仅处理任务元数据和调度状态

数据面（用户可完全自备）：
  Git 仓库 + 部署服务器 + CI/CD + 数据库
  → 源码、构建产物、业务数据在用户侧

基础设施管理采用资源池 + 环境组两层模型：
  资源池：所有注册的节点（Linux/Windows/K8s/GitLab 等）
       ↓ 按需选取
  环境组：从资源池中组合若干节点，形成逻辑环境
       ↓ 关联
  项目 ：使用环境组中标记为相应角色的节点

节点角色：agent（编码 Agent 运行）、deploy（部署目标）、ollama（AI 推理服务）
同一台服务器在不同环境组中可以有不同角色。

共享基础设施服务：
  Ollama（本地 AI 推理 + Embedding）：
    → 作为 InfraNode 注册，角色为 ollama
    → 系统启动时自动发现并配置 OLLAMA_BASE_URL
    → 多套系统可共用同一个 Ollama 实例
    → 零 token 成本的 embedding 计算 + 本地代码生成

GitLab 等外部依赖也作为节点统一管理，
便于 SaaS 场景下灵活切换平台提供或用户自带。
```

详细说明参见 [19-USER_MANUAL.md](19-USER_MANUAL.md) 第 11.3 节「基础设施」。
控制面/数据面分离的 SaaS 架构参见 [22-SAAS_ARCHITECTURE.md](22-SAAS_ARCHITECTURE.md)。

### 1. 模型代理池 - 统一管理，灵活分配

```
模型供应商配置存在 DB（model_providers 表）：
  ┌──────────────┬────────────────────────┬──────────────────┐
  │ 供应商名称    │ API Base               │ 支持的模型        │
  ├──────────────┼────────────────────────┼──────────────────┤
  │ deepseek     │ api.deepseek.com       │ deepseek-chat    │
  │ 147api       │ api.147api.com/v1      │ opus, sonnet     │
  │ anthropic    │ api.anthropic.com      │ opus, sonnet     │
  └──────────────┴────────────────────────┴──────────────────┘

任务分配时的模型解析链：
  任务指定模型 → 角色默认模型 → 供应商路由 → 实际调用

可通过 Web UI 管理供应商，随时切换/新增代理平台。
```

### 2. Agent 项目级隔离

```
每个 OpenClaw 实例绑定一个项目：
  - 项目创建 → 按需启动 OpenClaw 容器
  - 项目进行 → 容器持续运行，接受任务
  - 项目归档 → 备份 workspace → 停止容器
  - 项目恢复 → 新建容器 → 导入备份 → 继续工作

不同项目的 Agent 互不干扰，记忆体隔离。
```

### 3. 统一备份管理

```
调度器统一管理所有备份（Agent 不自己备份自己）：

备份内容：
  - workspace 目录（代码 + .openclaw/memory 记忆体）
  
备份存储：
  backups/{project_id}/{agent_id}/{timestamp}.tar.gz

备份触发：
  - 手动：通过 API 或 Web UI
  - 项目归档时：自动备份所有 Agent
  
恢复流程：
  - 新建 OpenClaw 容器
  - 从备份还原 workspace
  - 挂载 skills 和知识库
  - Agent 继续工作（记忆体完整）
```

### 4. 知识管理

```
知识源（五大来源）：
  ├── Document（阶段文档）     — 用户和 AI 生成的项目文档
  ├── Experience（经验库）     — 历史问题的解决方案（全局共享）
  ├── ProjectAsset（代码分析） — 上传代码的分析结果
  ├── TaskDocument（过程文档） — 任务执行中的产出
  └── docs/（项目设计文档）    — 系统自身的架构和设计文档

统一检索工具（knowledge_search.py）：
  三种模式：关键字精确 | tsvector 全文 | 语义向量搜索
  向量搜索：本地 Ollama bge-m3（零 token）→ 云端 fallback

多维度分类体系：
  ├── stage     阶段（0-7）
  ├── category  类型（plan/design/task/test/deploy...）
  ├── tags      标签（GIN 倒排索引，["数据库","API"]）
  └── status    生命周期（draft → approved）

AI 按需加载：
  [NEED_CONTEXT:key] 精确加载知识块
  [SEARCH:关键词]    模糊搜索所有知识源
```

详细设计参见 [07-KNOWLEDGE_SYSTEM.md](07-KNOWLEDGE_SYSTEM.md)。

## Leader 的职责边界

Leader 做什么：
- 理解需求，生成设计方案
- 将方案分解为 0.3-1h 粒度的任务
- 根据任务类型分配给合适的工程师
- 审查工程师提交的代码
- 做技术决策
- 向你汇报进度
- 管理模型供应商配置
- 管理项目生命周期（创建/归档/恢复/销毁）
- 统一备份管理

Leader 不做什么：
- 不写复杂业务代码
- 不排查复杂 Bug（交给工程师或升级给你）
- 不重写实现

## 工程师（OpenClaw）的自主性

每个 OpenClaw 实例是有独立思考能力的工程师：
- 接受任务后自主规划实现方案
- 能独立解决开发中遇到的问题
- 遇到超出能力范围的困难会主动上报
- 有自己的持久记忆，记住项目上下文和编码偏好
- 通过 Git 提交工作成果
- 绑定到特定项目，换项目是全新实例

## 技术选型

| 组件 | 技术 | 理由 |
|------|------|------|
| Leader 后端 | Python FastAPI | 异步、轻量、生态好 |
| Leader AI | DeepSeek API | 便宜、中文好、结构化输出强，Leader 不需要编码能力 |
| 模型代理 | OpenAI 兼容接口 | 所有代理平台都兼容，统一 SDK |
| 数据库 | PostgreSQL + pgvector | JSON/JSONB + 向量搜索、异步性能强、并发安全 |
| 本地 AI 推理 | Ollama | 零 token 成本的 embedding + 代码生成，CPU 即可运行 |
| Embedding | bge-m3 (via Ollama) | 中文效果优秀，568M 参数，纯 CPU 运行 |
| 工程师运行时 | OpenClaw Docker | 开源、自托管、模型无关、skill 可扩展 |
| 架构师模型 | Claude Opus（via 代理） | 编码理解最强，做设计决策和代码审查 |
| 工程师模型 | Claude Sonnet（via 代理） | 编码能力强，性价比高 |
| 版本控制 | Git | 工程师通过 Git 协作 |
| 通信 | HTTP Webhook + WebSocket | 调度器与 OpenClaw 之间、调度器与前端之间 |
| 前端 | Vue 3 + Vite | 独立 SPA，组件化设计 |
| 缓存 | Redis | 消息队列、会话缓存、实时通知 |

## 代码组织原则

从旧项目吸取教训——旧项目 app.py 2300 行、Sidebar.vue 2400 行，维护困难。
新项目从一开始就严格组件化：

后端：
- 每个路由模块独立文件（projects.py, tasks.py, agents.py, providers.py, webhook.py）
- 业务逻辑在 Service 层，路由层只做参数校验和转发
- AI 能力独立封装（ai_leader.py 不混入路由逻辑）
- 模型调用统一走 model_pool.py
- 备份管理独立封装（backup.py）

前端：
- 每个组件职责单一，不超过 300 行
- 页面级组件组合小组件，不堆砌逻辑
- 状态管理用 composables（useProjects, useStages, useTasks, useWebSocket）
- 通用 UI 组件独立（Toast, Modal, Progress 等）

## 成本策略

```
Leader（DeepSeek）：~¥1/百万token → 几乎免费
架构师（Opus via 代理）：官方5折 → $7.5/$37.5 每百万token
工程师（Sonnet via 代理）：官方5折 → $1.5/$7.5 每百万token

一个中等项目估算：
  Leader：~¥5（约$0.7）
  架构师：~$5-10
  工程师（3个）：~$10-20
  总计：~$15-30（约¥100-200）
```
